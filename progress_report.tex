\begin{abstract}
This progress report documents the development of a Weighted Randomized Kaczmarz solver designed to handle extremely large, sparse, underdetermined linear systems. Our target application is cone-beam CT reconstruction, involving binary projection matrices with $n = 512^3 \approx 134$ million columns and up to $m \approx 124$ million rows, where explicit matrix storage is infeasible ($\sim$16 petabytes dense). We present the algorithm design, implementation details, development iterations, integration with real CT scanner geometry from \texttt{mergecode.c}, performance optimizations including NumPy-vectorized ray tracing and a fast direct Kaczmarz loop with live progress tracking, and instructions for executing the reconstruction on \texttt{sim\_512.dat}.
\end{abstract}

\section{Introduction}
%==============================================================================

\subsection{Problem Statement}

We seek to solve an extremely large, sparse, underdetermined system of linear equations:
\begin{equation}
    Ax = b
\end{equation}
where:
\begin{itemize}
    \item $A \in \{0, 1\}^{m \times n}$ is a binary (bit) matrix
    \item $m < n$ (underdetermined system)
    \item $n = 512^3 = 134{,}217{,}728$ columns
    \item $m = 473 \times 512 = 242{,}176$ rows
    \item Explicit storage of $A$ is impossible ($\sim$64 TB for dense storage)
    \item Only row-wise access to indices of non-zero (1) entries is available
\end{itemize}

The key challenge is that traditional methods requiring explicit matrix access (LU decomposition, QR factorization, conjugate gradients) are infeasible at this scale. We required an algorithm that:
\begin{enumerate}
    \item Operates with \textbf{implicit matrix access} (row-by-row queries only)
    \item Scales to \textbf{hundreds of millions of unknowns}
    \item Converges to the \textbf{minimum-norm solution} for underdetermined systems
    \item Handles \textbf{potentially inconsistent systems} gracefully
\end{enumerate}

\subsection{The Kaczmarz Algorithm}

The classical Kaczmarz algorithm, also known as the Algebraic Reconstruction Technique (ART) in computed tomography, is an iterative row-action method for solving linear systems. Given a current iterate $x^{(k)}$, it projects onto the hyperplane defined by the $i$-th equation:
\begin{equation}
    x^{(k+1)} = x^{(k)} + \frac{b_i - a_i^\top x^{(k)}}{\|a_i\|^2} a_i
\end{equation}
where $a_i$ is the $i$-th row of $A$.

\subsubsection{Randomized Kaczmarz (Strohmer \& Vershynin, 2009)}

Instead of cycling through rows deterministically, we select row $i$ with probability proportional to $\|a_i\|^2$:
\begin{equation}
    P(i = k) = \frac{\|a_k\|^2}{\|A\|_F^2}
\end{equation}

This randomized variant achieves expected linear convergence:
\begin{equation}
    \mathbb{E}\left[\|x^{(k)} - x^*\|^2\right] \leq \left(1 - \frac{\sigma_{\min}^2(A)}{\|A\|_F^2}\right)^k \|x^{(0)} - x^*\|^2
\end{equation}

\subsubsection{Minimum-Norm Solution for Underdetermined Systems}

For underdetermined systems ($m < n$), infinitely many solutions exist. By initializing $x^{(0)} = 0$, the Kaczmarz iterates converge to the \textbf{minimum-norm solution}:
\begin{equation}
    x^* = A^\top(AA^\top)^{-1}b = A^\dagger b
\end{equation}
This is the unique solution with smallest $\ell_2$ norm, lying entirely in the row space of $A$.

\subsection{Implementation Plan}

Our implementation plan consisted of the following phases:
\begin{enumerate}
    \item \textbf{Phase 1:} Core algorithm implementation with basic functionality
    \item \textbf{Phase 2:} Efficient weighted sampling using Walker's Alias Method
    \item \textbf{Phase 3:} Implicit matrix abstraction for memory efficiency
    \item \textbf{Phase 4:} Adaptive stopping criteria and convergence detection
    \item \textbf{Phase 5:} Testing, validation, and performance optimization
\end{enumerate}

\section{Development Iterations}
%==============================================================================

\subsection{Iteration 1: Naive Prototype}

\textbf{Goal:} Establish a working baseline implementation.

\textbf{Approach:} We began with a straightforward implementation that explicitly stored the matrix and used simple cyclic row selection.

\textbf{Issues Encountered:}
\begin{itemize}
    \item Memory consumption was prohibitive even for moderate test cases
    \item Cyclic row selection showed erratic convergence behavior
    \item No mechanism to detect when the system was inconsistent
\end{itemize}

\textbf{Lessons Learned:} The need for implicit matrix representation and probabilistic row selection became immediately apparent.

\subsection{Iteration 2: Randomized Row Selection}

\textbf{Goal:} Implement the randomized Kaczmarz variant with convergence guarantees.

\textbf{Approach:} We implemented weighted random sampling where rows are selected with probability proportional to their squared norm. For our binary matrices:
\begin{equation}
    \|a_i\|^2 = \text{nnz}(a_i) = \text{number of 1s in row } i
\end{equation}

\textbf{Initial Implementation:} Used binary search on cumulative probability distribution.

\textbf{Issues Encountered:}
\begin{itemize}
    \item $O(\log m)$ sampling overhead became significant over millions of iterations
    \item Memory allocation for cumulative arrays was inefficient
\end{itemize}

\textbf{Resolution:} Migrated to Walker's Alias Method for $O(1)$ sampling.

\subsection{Iteration 3: Walker's Alias Method Integration}

\textbf{Goal:} Achieve constant-time weighted sampling.

\textbf{Approach:} Implemented Walker's Alias Method which provides:
\begin{itemize}
    \item $O(m)$ preprocessing time
    \item $O(m)$ space for probability and alias tables
    \item $O(1)$ sampling time per iteration
\end{itemize}

\textbf{Outcome:} Significant performance improvement, reducing per-iteration overhead to a constant regardless of the number of rows.

\subsection{Iteration 4: Implicit Matrix Abstraction}

\textbf{Goal:} Enable solving systems where the matrix cannot be explicitly stored.

\textbf{Approach:} Designed an abstraction layer with two implementations:
\begin{enumerate}
    \item \texttt{ImplicitBitMatrix}: Accepts a row-access function that returns column indices of non-zeros
    \item \texttt{ExplicitSparseRowMatrix}: For smaller test cases where explicit storage is feasible
\end{enumerate}

\textbf{Key Design Decision:} The matrix interface requires only:
\begin{itemize}
    \item Dimensions $(m, n)$
    \item Row access: $\text{get\_row\_indices}(i) \rightarrow \{j : A_{ij} = 1\}$
    \item Row weights (precomputed during initialization)
\end{itemize}

\textbf{Memory Achievement:} Solution vector requires $\sim$1.07 GB; matrix is never materialized.

\subsection{Iteration 5: Convergence Detection}

\textbf{Goal:} Implement robust stopping criteria with inconsistency detection.

\textbf{Approach:} Designed a moving-window approach monitoring squared residuals:

\textbf{Convergence Criterion:}
\begin{equation}
    \text{mean}(H) < \epsilon^2 \implies \text{CONVERGED}
\end{equation}
where $H$ is a window of recent squared residuals.

\textbf{Stagnation Detection:} For inconsistent systems:
\begin{equation}
    \text{stagnation} = \left(\frac{\sigma(H)}{\mu(H)} < \tau_{cv}\right) \land \left(\mu(H) > \epsilon^2\right)
\end{equation}

\textbf{Outcome:} The solver now gracefully handles inconsistent systems, providing useful approximate solutions rather than diverging.

\subsection{Iteration 6: Performance Tuning and Testing}

\textbf{Goal:} Validate correctness and optimize for production use.

\textbf{Activities:}
\begin{itemize}
    \item Implemented comprehensive test suite
    \item Verified minimum-norm property on underdetermined systems
    \item Profiled and optimized critical code paths
    \item Added verbosity controls and progress tracking
\end{itemize}

\subsection{Iteration 7: Cone-Beam CT Integration}

\textbf{Goal:} Integrate the solver with real cone-beam CT scanner geometry from \texttt{mergecode.c} and enable reconstruction from \texttt{sim\_512.dat}.

\textbf{Approach:} We translated the C-language ray-tracing routine (\texttt{weightgen()}) and projection geometry from \texttt{mergecode.c} into a Python module (\texttt{examples/ct\_reconstruction.py}). The key components are:
\begin{itemize}
    \item \texttt{CTGeometry}: Encapsulates all scanner parameters---source-object distance (166.641\,mm), source-detector distance (1238.157\,mm), 640° angular span, 473 projections, $512 \times 512$ detector, 0.8\,mm pixel size, and 80.86\,mm phantom diameter.
    \item \texttt{ImplicitCTMatrix}: Wraps the geometry as an implicit matrix compatible with the Kaczmarz solver. Each row corresponds to a single detector measurement, and \texttt{get\_row\_indices()} computes on-the-fly which voxels a ray intersects using the cone-beam geometry.
    \item \texttt{compute\_ray\_voxels()}: Vectorized ray-voxel intersection function that converts a (projection, slice, detector) triplet into the list of voxel indices the ray passes through.
\end{itemize}

\textbf{Full-scale matrix dimensions:}
\begin{itemize}
    \item Rows $m = 473 \times 512 \times 512 = 123{,}994{,}112$ (one row per detector pixel per projection)
    \item Columns $n = 512^3 = 134{,}217{,}728$ (one column per voxel)
    \item Dense storage: $\sim$16 petabytes---completely infeasible
    \item Our implicit representation: $\sim$1.07\,GB (solution vector only)
\end{itemize}

\textbf{Outcome:} Successful end-to-end reconstruction pipeline: load phantom $\rightarrow$ forward-project sinogram $\rightarrow$ reconstruct via Kaczmarz.

\subsection{Iteration 8: Performance Optimization}

\textbf{Goal:} Improve execution speed, add dynamic terminal progress, and provide time-to-completion estimates.

\textbf{Optimizations implemented:}
\begin{enumerate}
    \item \textbf{Vectorized ray tracing.} The original C code loops over each pixel intersection sequentially. We replaced this with NumPy vectorized operations: all $n_{\text{pix}}$ intersection points are computed in a single array operation, with boolean masking for in-volume filtering. This eliminates per-pixel Python loop overhead.
    
    \item \textbf{Fast direct Kaczmarz loop (\texttt{kaczmarz\_direct()}).} We implemented an optimised inline Kaczmarz loop that bypasses the generic solver infrastructure. Key differences from the general-purpose solver:
    \begin{itemize}
        \item Direct \texttt{np.random.RandomState.randint()} for row sampling (bypasses Walker's Alias tables, which are expensive to build for 124M rows)
        \item Circular buffer for moving-window residual tracking (no deque overhead)
        \item Convergence checked every \texttt{window\_size} iterations instead of every iteration
        \item All operations inlined to minimise function-call overhead
    \end{itemize}
    
    \item \textbf{Dynamic progress bar (\texttt{ProgressBar} class).} A single-line terminal progress bar that updates in-place using carriage returns. Features:
    \begin{itemize}
        \item Throttled output (50\,ms minimum interval) to avoid I/O bottlenecks
        \item Processing rate (iterations/second)
        \item Elapsed time and estimated time remaining (ETA)
        \item Visual bar with filled/empty characters
    \end{itemize}
    
    \item \textbf{LRU caching.} \texttt{ImplicitCTMatrix} uses \texttt{functools.lru\_cache} to cache recently computed rows, avoiding redundant ray-tracing for repeatedly sampled rows.
\end{enumerate}

\textbf{Benchmark results} (32\textsuperscript{3} demo, 20 projections, 100K iterations):
\begin{itemize}
    \item Forward projection: 20,480 rays at $\sim$15,500 rays/s
    \item Kaczmarz reconstruction: $\sim$22,500 iterations/s
    \item Total time: $\sim$5.5 seconds (including phantom creation)
\end{itemize}

\subsection{Algorithm Design}

Our final algorithm (Algorithm~\ref{alg:kaczmarz}) incorporates all design requirements:

\begin{algorithm}[h]
\caption{Weighted Randomized Kaczmarz with Adaptive Stopping}
\label{alg:kaczmarz}
\begin{algorithmic}[1]
\Require Row access function $\text{row}(i)$, RHS $b \in \mathbb{R}^m$, tolerance $\epsilon$, window $W$
\Ensure Approximate minimum-norm solution $x \in \mathbb{R}^n$
\State $x \gets \mathbf{0} \in \mathbb{R}^n$ \Comment{Initialize for minimum-norm}
\State Compute row weights $w[i] \gets |\text{row}(i)|$ for all $i$
\State Total weight $S \gets \sum_i w[i]$
\State Build alias sampler with $P[i] = w[i]/S$
\State Initialize residual history $H \gets$ empty deque of size $W$
\For{$k = 1, 2, \ldots$}
    \State Sample row index $i \sim P$ \Comment{$O(1)$ with alias method}
    \State $J \gets \text{row}(i)$ \Comment{Non-zero indices}
    \State $\langle a_i, x \rangle \gets \sum_{j \in J} x[j]$ \Comment{Sparse dot product}
    \State $r_i \gets b_i - \langle a_i, x \rangle$ \Comment{Residual}
    \State $\alpha \gets r_i / |J|$ \Comment{Step size (since $\|a_i\|^2 = |J|$)}
    \State $x[j] \gets x[j] + \alpha$ for all $j \in J$ \Comment{Update}
    \State Append $|r_i|^2$ to $H$
    \If{$|H| = W$}
        \If{$\text{mean}(H) < \epsilon^2$} \Return $x$ \Comment{Converged}
        \EndIf
        \If{stagnation detected} \Return $x$ \Comment{Inconsistent system}
        \EndIf
    \EndIf
\EndFor
\end{algorithmic}
\end{algorithm}

\subsection{Performance Considerations}

For the full $512^3$ problem:
\begin{itemize}
    \item The system has $n = 134{,}217{,}728$ unknowns and up to $m = 124$ million measurements.
    \item Forward projection with 50 projections traces $50 \times 512 \times 512 = 13{,}107{,}200$ rays.
    \item Reconstruction time depends heavily on the number of iterations. At $\sim$20{,}000 iter/s, 500K iterations takes roughly 25 seconds of Kaczmarz time.
    \item The forward projection phase is typically the bottleneck for large numbers of projections.
    \item Increasing \texttt{--nproj} improves reconstruction quality but increases forward projection time linearly.
\end{itemize}

\section{Future Work}
%==============================================================================

Potential areas for future enhancement include:
\begin{enumerate}
    \item \textbf{Block Kaczmarz:} Process multiple rows per iteration for improved convergence
    \item \textbf{Parallelization:} Distributed computation for even larger systems
    \item \textbf{Acceleration:} Momentum-based variants (Heavy Ball, Nesterov)
    \item \textbf{Preconditioning:} Row scaling for improved conditioning
    \item \textbf{GPU Implementation:} Leverage parallel architectures for sparse updates
    \item \textbf{Multi-resolution Reconstruction:} Start with a coarse volume and progressively refine
    \item \textbf{Regularization:} Total-variation or sparsity penalties for improved image quality
\end{enumerate}
